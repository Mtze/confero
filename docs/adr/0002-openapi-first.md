# ADR 0002 - OpenAPI-first API design

**Status:** Accepted

## Context

A common failure mode in API-driven projects is drift: the implementation
diverges from the documented contract, and the frontend or external callers
start depending on undocumented behavior. This is especially problematic when
an AI agent is one of the primary API callers (FR-18, bulk import).

We need a single source of truth for the API shape.

## Decision

`api/openapi.yaml` is the single source of truth for the HTTP API.

Rules (enforced by tooling):
1. Any API change starts with editing `api/openapi.yaml`.
2. Go server stubs are generated by `oapi-codegen` into `server/internal/api/`
   (generated; never edited by hand).
3. The TypeScript client is generated by `@hey-api/openapi-ts` into
   `web/src/api/` (generated; never edited by hand).
4. The Bruno collection (`bruno/confero/`) has one `.bru` request per
   OpenAPI operation. CI fails if any operation is missing a Bruno request.
5. CI runs `make generate` after every build and fails if the working tree is
   dirty (catches stale generated code before it is merged).
6. `redocly lint api/openapi.yaml` runs in CI and must pass.

## Consequences

- Every API consumer (Go handlers, TypeScript SPA, Bruno, LLM agents) works
  from the same schema, eliminating an entire class of integration bugs.
- Adding a field requires touching the spec first, which makes the change
  visible in code review before any implementation lands.
- The code-generation step adds a small complexity cost: contributors need
  `oapi-codegen` and `@hey-api/openapi-ts` installed locally (or can rely on
  CI). `make install-tools` handles this.
- The strict "no hand-edits to generated code" rule means some ergonomics
  belong in the generation config (`oapi-codegen.yaml`), not in the output.

Locks: REQUIREMENTS NFR-2, D2.
